# 校园二手交易平台规范目标文档

## 1. 项目概述

### 1.1 项目名称

校园二手交易平台

### 1.2 项目定位

面向校内学生用户，提供一个可信、便捷、低门槛的二手物品发布、浏览、沟通与交易平台。项目采用前后端分离架构，支持后续扩展为 Web 管理端、移动端 H5、小程序或 App。

### 1.3 项目目标

本项目的核心目标如下：

1. 为学生提供一个简单易用的校内二手交易渠道。
2. 降低二手商品信息不透明、沟通效率低、交易信任不足等问题。
3. 建立可扩展的前后端分离系统，为后续增加支付、担保、认证、推荐等能力预留空间。
4. 形成一套可执行、可分阶段交付的开发规范，便于按计划推进实现。

### 1.4 项目原则

项目实施遵循以下原则：

1. 先完成核心闭环，再逐步扩展功能。
2. 优先保证平台可用性、稳定性和安全性。
3. 优先满足校园场景下的真实交易需求。
4. 接口、数据、权限设计规范化，方便多人协作和后期维护。

## 2. 建设范围

### 2.1 本期建设范围

本期优先完成平台最小可用版本，即用户可以完成从注册登录到商品发布、商品浏览、私信沟通、订单确认的基础交易流程。

### 2.2 核心业务闭环

平台必须支持以下完整闭环：

1. 用户注册/登录。
2. 用户完善个人资料。
3. 用户发布闲置商品。
4. 其他用户浏览、搜索、筛选商品。
5. 买卖双方进行站内沟通。
6. 卖家确认成交或下架商品。
7. 系统记录交易状态和基础行为数据。

### 2.3 暂不纳入本期范围

以下功能不作为第一阶段必做项：

1. 在线支付与资金担保。
2. 物流系统对接。
3. 智能推荐算法。
4. 复杂营销系统。
5. 跨校区联盟交易。
6. 多语言支持。

## 3. 目标用户与角色定义

### 3.1 目标用户

1. 校内有闲置物品出售需求的学生。
2. 校内有低价购买需求的学生。
3. 平台运营与审核人员。

### 3.2 角色定义

#### 普通用户

具备注册、登录、浏览商品、发布商品、收藏商品、发起咨询、下单确认、查看订单、举报内容等能力。

#### 管理员

具备用户管理、商品审核、举报处理、类目配置、公告管理、交易数据查看、违规内容下架等能力。

## 4. 业务目标

### 4.1 用户目标

1. 用户能在 3 分钟内完成注册并发布第一件商品。
2. 用户能在 30 秒内通过搜索或筛选找到目标商品。
3. 用户能快速联系卖家并确认交易状态。

### 4.2 平台目标

1. 提供稳定的商品信息发布和展示能力。
2. 建立基本的校内身份可信体系。
3. 支持基础的内容审核与违规处理。
4. 支持后续功能扩展和业务增长。

### 4.3 技术目标

1. 前后端彻底分离，接口风格统一。
2. 数据模型清晰，便于后续演进。
3. 权限控制明确，避免越权访问。
4. 系统具备基础日志、监控和异常处理能力。

## 6. 非功能性要求

### 6.1 性能要求

1. 首页商品列表接口响应时间目标不高于 1 秒。
2. 常规查询接口响应时间目标不高于 500 毫秒到 1500 毫秒。
3. 支持至少 1000 名注册用户的初期规模。

### 6.2 安全要求

1. 密码必须加密存储，禁止明文保存。
2. 必须进行身份认证和接口鉴权。
3. 必须校验上传文件类型和大小。
4. 必须防止常见安全风险，如 SQL 注入、XSS、CSRF、越权访问。
5. 关键操作必须记录审计日志。

### 6.3 可用性要求

1. 页面操作路径清晰，核心流程简单。
2. 重要操作提供提示与确认机制。
3. 异常状态需要有明确反馈。

### 6.4 可维护性要求

1. 接口命名统一。
2. 代码分层清晰。
3. 文档保持同步更新。
4. 具备基础单元测试和接口测试能力。

### 6.5 可扩展性要求

系统架构必须支持后续扩展以下能力：

1. 第三方登录。
2. 小程序端。
3. 移动端 App。
4. 支付系统。
5. 智能推荐。
6. 信用评价体系。

## 7. 技术方案建议

### 7.1 总体架构

建议采用前后端分离架构：

1. 前端负责页面渲染、交互逻辑、状态管理和接口调用。
2. 后端负责业务逻辑、权限控制、数据存储、文件上传和消息处理。
3. 管理端可单独部署，也可与用户端共享同一后端服务。

### 7.2 前端建议

建议技术选型如下：

1. Web 前端：Vue 3 + Vite + TypeScript + Pinia + Vue Router + Axios。
2. 管理端：Vue 3 + Element Plus。
3. 样式方案：SCSS 或 Tailwind CSS。

前端目录建议：

1. `src/api`：接口封装。
2. `src/views`：页面模块。
3. `src/components`：公共组件。
4. `src/stores`：状态管理。
5. `src/router`：路由管理。
6. `src/utils`：工具函数。

### 7.3 后端建议

可选两种主流方案，任选一种即可：

方案 A：

1. Java Spring Boot + MySQL + Redis + MinIO。

方案 B：

1. Node.js NestJS + MySQL + Redis + MinIO。

后端分层建议：

1. Controller 层：处理请求与响应。
2. Service 层：封装业务逻辑。
3. Repository/Mapper 层：数据访问。
4. Common 层：公共组件、异常、工具、响应封装。

### 7.4 数据库建议

关系型数据库建议使用 MySQL。

缓存建议：

1. Redis 用于验证码、登录态、热点数据缓存。

文件存储建议：

1. 商品图片使用对象存储或本地静态资源服务。

## 9. 接口规范要求

### 9.1 接口设计原则

1. 接口统一使用 RESTful 风格。
2. 接口返回结构保持一致。
3. 统一处理错误码与错误信息。
4. 重要接口必须鉴权。

### 9.2 返回结构建议

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 9.3 接口模块划分建议

1. `/api/auth`：认证相关接口。
2. `/api/users`：用户相关接口。
3. `/api/products`：商品相关接口。
4. `/api/categories`：分类相关接口。
5. `/api/favorites`：收藏相关接口。
6. `/api/messages`：消息相关接口。
7. `/api/orders`：交易相关接口。
8. `/api/reports`：举报相关接口。
9. `/api/admin/*`：管理端接口。

### 9.4 接口文档要求

1. 必须提供接口说明。
2. 必须提供请求参数定义。
3. 必须提供返回字段定义。
4. 必须提供错误码说明。
5. 推荐使用 Swagger/OpenAPI 自动生成接口文档。

## 10. 权限与审核规范

### 10.1 权限控制

1. 普通用户仅能操作自己的商品、消息、订单等数据。
2. 管理员具备平台管理权限，但应区分超级管理员和普通运营管理员。
3. 所有敏感操作必须做权限校验。

### 10.2 审核规则

平台必须具备基础审核规则：

1. 禁止发布违规、违法、诈骗类内容。
2. 禁止发布虚假价格或恶意引流内容。
3. 禁止发布非校园场景不适合的商品信息。
4. 举报内容必须进入后台待处理列表。

## 14. 风险与应对

### 14.1 需求变动风险

应对：
先明确 MVP 范围，新增需求放入后续迭代，避免首期范围失控。

### 14.2 技术复杂度风险

应对：
第一阶段避免引入支付、推荐、复杂 IM 等高复杂度能力。

### 14.3 数据安全风险

应对：
强化认证、鉴权、日志与内容审核机制。

### 14.4 项目进度风险

应对：
按模块拆解任务，以“先核心、后增强”的方式推进。
