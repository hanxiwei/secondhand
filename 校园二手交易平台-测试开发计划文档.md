# 校园二手交易平台测试开发计划文档

## 1. 文档说明

### 1.1 文档目的

本文件用于规划“校园二手交易平台全链路质量保障（功能 & 自动化）”测试开发实践的建设目标、实施范围、技术方案、阶段安排和交付物，帮助项目从“可用”进一步走向“可测、可回归、可持续集成”。

### 1.2 适用范围

本计划适用于当前校园二手交易 Web 平台的测试开发工作，覆盖以下核心业务范围：

1. 用户注册、登录、登录态校验。
2. 商品发布、列表、搜索、详情浏览。
3. 收藏、消息会话、订单创建与状态流转。
4. 关键接口稳定性、前端页面关键路径回归。
5. 自动化测试执行、测试报告生成、质量门禁与通知联动。

### 1.3 项目定位

本项目不再只关注“功能开发是否完成”，而是进一步引入测试开发思维，建设一套适合小型 Web 项目的轻量级质量保障体系。目标是在有限的人力和周期下，通过接口自动化、UI 自动化、AI 报告分析和 CI/CD 集成，形成“需求实现后可快速验证、迭代发布前可自动回归、失败后可快速定位”的测试闭环。

## 2. 项目背景

### 2.1 背景说明

校园二手交易平台属于典型的多角色、多流程 Web 项目，既包含用户端的注册登录、商品浏览、沟通和下单，也包含卖家端的发布商品、处理订单和管理信息。随着功能逐步增多，传统手工点点点的回归方式容易出现以下问题：

1. 功能迭代快，手工回归成本高。
2. 登录、搜索、下单等核心流程每次都要重复验证，效率低。
3. 问题出现后需要人工翻日志和页面复现，定位慢。
4. 小型项目通常没有成熟测试平台，测试能力难以工程化沉淀。

### 2.2 当前痛点

结合本项目现状，测试开发阶段重点解决以下三类问题：

1. 回归人力不足：每次改动后都靠人工完整回归，耗时大且容易遗漏。
2. 结果分析滞后：测试执行后缺少统一报告和自动归因，发布决策慢。
3. 质量门禁缺失：代码通过构建不代表功能无风险，缺少自动化放行标准。

## 3. 建设目标

### 3.1 总体目标

围绕校园二手交易平台构建“接口自动化 + UI 自动化 + AI 分析 + CI 门禁”的轻量级测试开发体系，在 2025.08 至 2025.10 周期内完成基础落地。

### 3.2 分项目标

1. 建立统一的接口自动化测试框架，覆盖登录、商品、搜索、订单三大核心域。
2. 建立基础 UI 自动化回归能力，覆盖首页搜索、商品详情、下单关键路径。
3. 接入 Allure 报告，沉淀测试执行结果、失败截图和日志。
4. 增加 AI 摘要分析脚本，对失败结果进行自动分类和结构化输出。
5. 将自动化测试集成进 Jenkins 流水线，形成“提交即测、定时回归、失败告警”的持续测试机制。
6. 建立最小可用质量门禁，支撑版本回归和毕业设计答辩演示。

## 4. 测试范围与优先级

### 4.1 核心测试范围

#### P0 核心链路

1. 注册、登录、获取当前用户信息。
2. 商品发布、商品列表、搜索、商品详情。
3. 买家发起购买意向、卖家确认、卖家完成订单。
4. 登录失效、未登录访问拦截、错误提示展示。

#### P1 高频功能

1. 收藏与取消收藏。
2. 发起会话、发送消息、查看会话列表。
3. 个人中心资料查询与编辑。
4. 商品上下架、删除、我的订单、清空已取消订单。

#### P2 扩展质量项

1. 接口异常输入校验。
2. 边界值验证，如金额、库存、字段长度。
3. 页面空状态、失败提示、图片兜底、权限跳转。

### 4.2 暂不纳入首期范围

1. 大规模性能压测。
2. 复杂安全渗透测试。
3. 多浏览器矩阵兼容性全覆盖。
4. 容器化大规模并行执行平台。

## 5. 技术方案设计

### 5.1 分层测试架构

本项目采用“接口层 + UI 层 + 报告分析层 + 流水线层”的分层设计。

1. 接口层负责快速验证业务逻辑、状态码、响应体和异常分支。
2. UI 层负责覆盖真实用户路径，验证页面交互和流程打通情况。
3. 报告分析层负责解析测试结果、分类失败原因、输出摘要。
4. 流水线层负责触发执行、卡口控制和消息通知。

### 5.2 接口自动化方案

技术选型如下：

1. `Python`
2. `pytest`
3. `requests`
4. `PyYAML`
5. `allure-pytest`

核心建设内容：

1. 封装统一 `Session` 会话管理，支持 Token 自动注入。
2. 封装基础请求方法、公共断言方法、响应校验方法。
3. 使用 `YAML/CSV` 维护测试数据和异常场景矩阵。
4. 支持正常场景、异常场景、鉴权场景和边界值场景。

### 5.3 UI 自动化方案

技术选型如下：

1. `Python`
2. `pytest`
3. `Playwright`
4. `playwright-pytest`
5. `Page Object Model (POM)`

核心建设内容：

1. 将登录页、首页、商品详情页、订单页封装为页面对象。
2. 实现搜索、查看详情、发起购买意向、订单确认等关键用例。
3. 对失败步骤自动截图并写入 Allure。
4. 基于 `Playwright` 的自动等待、定位器和多浏览器能力，提高脚本稳定性。
5. 将页面元素定位、动作方法、断言逻辑分层，降低后续维护成本。

### 5.4 AI 结果分析方案

技术选型如下：

1. `Python`
2. `JSON`
3. `正则表达式`
4. `规则引擎`

首期采用轻量方案：

1. 解析 Allure JSON 结果文件。
2. 解析 pytest 输出日志和错误堆栈。
3. 按关键词和规则分类失败原因。
4. 输出结构化摘要，分类为：
   1. 断言失败
   2. 服务端异常
   3. 超时问题
   4. 环境问题

### 5.5 CI/CD 集成方案

技术选型如下：

1. `Jenkins`
2. `Git`
3. `Allure Report`
4. `飞书机器人`

集成策略如下：

1. 支持代码提交触发。
2. 支持每日定时回归。
3. 接口自动化先执行，UI 自动化后执行。
4. 执行完成后自动生成报告并推送摘要。
5. 支持将执行结果推送到 `飞书群`，后续如有需要也可兼容 `钉钉机器人`。
6. 当通过率低于阈值或 P0 用例失败时自动阻断构建。

## 6. 测试设计方法

### 6.1 用例设计原则

本项目优先采用以下测试设计方法：

1. 等价类划分。
2. 边界值分析。
3. 场景法。
4. 状态迁移法。
5. 错误猜测法。

### 6.2 重点测试矩阵

重点围绕以下内容建立测试矩阵：

1. 登录场景：正常登录、密码错误、账号不存在、未填写字段、重复请求。
2. 搜索场景：空关键词、中文关键词、特殊字符、无结果返回。
3. 商品场景：必填项缺失、价格边界、图片上传、分类选择。
4. 订单场景：买家下单、卖家确认、卖家完成、买家/卖家取消、重复下单。
5. 权限场景：未登录访问、越权操作、自己购买自己的商品。

| 第 2 周  | 接口框架搭建       | 完成 pytest + requests 框架、环境配置、公共封装             |

## 9. 详细实施任务

### 9.1 第一阶段：测试分析与环境准备

1. 梳理现有接口清单和页面流程。
2. 确认测试环境地址、账号策略、测试数据准备方式。
3. 输出测试范围表、优先级表、风险清单。

### 9.2 第二阶段：接口自动化落地

1. 搭建基础项目目录。
2. 封装基础请求层与断言层。
3. 优先覆盖登录、用户、商品、订单接口。
4. 设计 `20+` 登录异常与边界场景。

### 9.3 第三阶段：UI 自动化落地

1. 抽取登录页对象、首页对象、商品详情页对象、订单页对象。
2. 使用 `Playwright + POM` 实现“搜索商品 -> 查看详情 -> 发起购买意向 -> 订单页校验”主链路脚本。
3. 增加失败截图与元素等待机制。

### 9.4 第四阶段：报告与分析能力建设

1. 接入 Allure 结果输出。
2. 编写日志解析与失败分类脚本。
3. 输出通过率、失败模块、典型报错、失败归因摘要。

### 9.5 第五阶段：流水线与门禁建设

1. 新建 Jenkins Job 或 Pipeline。
2. 配置代码提交触发与定时触发。
3. 增加门禁逻辑：
   1. 通过率低于 `90%` 阻断构建。
   2. P0 用例失败阻断构建。
4. 接入飞书机器人推送结果，并预留钉钉兼容能力。

## 11. 质量指标与验收标准

### 11.1 过程指标

1. 接口自动化覆盖核心接口不少于 `20` 个场景。
2. UI 自动化覆盖核心主流程不少于 `8` 条。
3. 测试报告可自动生成并可追溯。
4. 每次执行结束后可输出失败分类摘要。

### 11.2 结果指标

1. 核心流程自动化通过率达到 `90%` 及以上。
2. 关键回归执行时间相较手工方式显著降低。
3. P0 问题在发布前均可被自动化用例发现。
4. Jenkins 流水线可稳定执行自动化并推送通知。

## 12. 风险与应对策略

### 12.1 主要风险

1. 页面元素频繁变动，导致 UI 脚本维护成本增加。
2. 测试数据不稳定，导致订单和商品类用例互相干扰。
3. 本地环境不统一，影响执行结果一致性。
4. AI 失败归因规则初期不完善，存在误判。

### 12.2 应对措施

1. 优先稳定接口层，UI 层只覆盖关键主路径。
2. 引入独立测试账号和测试数据清理机制。
3. 统一环境配置，明确测试前置条件。
4. 先采用规则引擎方案，再逐步优化归因准确率。

## 14. 下一步实施建议

基于当前校园二手交易平台现状，建议按以下顺序立即进入实操：

1. 先补测试工程目录与基础依赖。
2. 优先实现登录、商品搜索、下单三条接口自动化。
3. 再补首页搜索、商品详情、订单页三条 UI 自动化主链路。
4. 最后接入 Allure、AI 分析脚本和 Jenkins 门禁。

以上顺序可以确保先把“能跑起来、能回归、能产出报告”的最小测试体系搭起来，再逐步升级为完整测试开发项目。
